# Проверка файла `opening_1step_reports_financial.md`

Ниже перечислены технические замечания и предлагаемые правки. Все предлагаемые формулировки вынесены в красные рамки.

---

## 1. Архитектура: DWH/витрина в исходном BRD указана как обязательный слой

**Где:** разделы 2.1–2.2.  
**Проблема:** в BRD указано: данные из 1С забираются в промежуточное хранилище (DWH/витрину), очистка и исключение внутригрупповых оборотов выполняются на стороне DWH. В текущем MD хранилище описано как «Временные таблицы / CSV / БД», а схема выглядит как прямой поток `1С → ETL → Битрикс24`. Это может создать у заказчика неверное ожидание, что DWH не нужен.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Заменить архитектурную схему на: <b>1С УТ 11.6 → ETL → DWH/витрина управленческих данных → Битрикс24</b>.<br><br>
В разделе компонентов заменить строку «Хранилище | Временные таблицы / CSV / БД» на:<br>
<b>Хранилище / витрина данных | PostgreSQL / MS SQL / иная согласованная БД | Промежуточное хранение очищенных, нормализованных и агрегированных данных; исключение внутригрупповых оборотов; контрольные сверки с 1С.</b>
</div>

## 2. COM-соединение с 1С зависит от Windows-инфраструктуры

**Где:** раздел 2.3, вариант A.  
**Проблема:** COM-соединение с 1С работает только в Windows-среде. Если ETL будет запускаться на Linux-сервере или в облаке, COM не подойдёт. Сейчас это не отмечено.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Уточнить способ интеграции с 1С на этапе обследования. Возможные варианты: COM-соединение только при запуске ETL на Windows; регламентная выгрузка из 1С в CSV/JSON/XML; HTTP/OData/web-сервис 1С; обмен через промежуточную БД. Финальный вариант выбирается после проверки доступа к базе 1С и политики безопасности заказчика.
</div>

## 3. Механизмы Битрикс24 описаны слишком уверенно

**Где:** разделы 4.2–4.3.  
**Проблема:** «Пользовательские таблицы (External Tables)» и «Отчёты (Report)» не стоит указывать как гарантированный стандартный механизм без проверки конкретной редакции Битрикс24. Для коробочного Битрикс24 обычно рассматривают highload-блоки, инфоблоки, списки, смарт-процессы, кастомный модуль или отдельную витрину с отображением через виджет. REST `batch` также не всегда подходит для массовой ETL-загрузки из-за лимитов.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Заменить таблицу механизмов Битрикс24 на предварительную:<br><br>
<b>Возможные механизмы хранения и отображения данных в Битрикс24: highload-блоки, инфоблоки, списки/смарт-процессы, кастомный модуль, отдельная DWH-витрина + виджет. Конкретный механизм выбирается после проверки редакции Битрикс24, доступных модулей, прав администратора, лимитов REST API и требований к объёму данных.</b>
</div>

## 4. Использование бухгалтерских счетов противоречит заявленному источнику «1С:УТ 11.6»

**Где:** разделы 3.2, 5.3, 7.3.

Но далее используются «оборот по счёту 44», «оборот по счёту 26», «план счетов бухгалтерского учёта», «ОСВ для баланса». Это объекты бухгалтерского учёта, которые могут отсутствовать или быть неполными в УТ. Если требуется бухгалтерская ОСВ, нужен доступ к 1С:Бухгалтерии или подтверждение, что эти данные есть в УТ.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>

<b>Источник требует проверки: если в 1С:УТ 11.6 нет полноценного бухгалтерского контура, показатели расходов, баланса, основных средств, кредитов и займов строятся по управленческим статьям, документам казначейства, взаиморасчётам и регистрам УТ. Если нужна бухгалтерская точность, подключается дополнительный источник — 1С:Бухгалтерия / бухгалтерская база.</b>
</div>

## 5. ОПиУ смешан с кассовым методом

**Где:** раздел 5.4.  
**Проблема:** строка «менеджерский результат считается по фактическому получению ДС» методологически ближе к кассовому отчёту, а не к классическому ОПиУ/P&L. Если ОПиУ считать по оплатам, он начнёт дублировать ОДДС и искажать доходность периода. В исходном списке отчётов этот пункт уже спорный: результат менеджера может считаться по оплате, но прибыльность бизнеса лучше считать по отгрузке/начислению.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Разделить два показателя:<br>
1. <b>ОПиУ/P&amp;L</b> — считается по отгрузке/начислению за период: выручка, себестоимость, валовая прибыль, расходы, EBITDA, чистая прибыль.<br>
2. <b>Менеджерский результат по оплатам</b> — отдельный KPI для мотивации и контроля: оплаченная выручка, оплаченная маржа, задолженность по сделкам.<br><br>
Фразу «менеджерский результат считается по фактическому получению ДС» перенести из методики ОПиУ в отдельное описание KPI отдела продаж/мотивации.
</div>

## 6. Формулы EBITDA, чистой прибыли и маржи требуют уточнения

**Где:** раздел 5.3.  
**Проблема:** EBITDA указана как `gross_profit - commercial_expenses - admin_expenses`. Это может быть корректно только если в расходах нет амортизации, процентов, налогов и прочих операционных статей. Формула чистой прибыли `ebitda - проценты - налоги` также неполная: из EBITDA обычно нужно вычесть амортизацию, проценты, налоги и учесть прочие доходы/расходы. Также в формулах процентов нет защиты от деления на ноль.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Уточнить формулы:<br>
<b>gross_margin_pct = gross_profit / revenue * 100</b>, если revenue &gt; 0, иначе 0 или null.<br>
<b>EBITDA = gross_profit - commercial_expenses - admin_expenses - other_operating_expenses + other_operating_income + depreciation_and_amortization</b> — либо явно указать, что это «упрощённая управленческая EBITDA».<br>
<b>net_profit = EBITDA - depreciation_and_amortization - interest_expense - taxes +/− other_non_operating_items</b>.
</div>

## 7. В ОДДС поле отклонения не имеет однозначной базы расчёта

**Где:** раздел 6.3.  
**Проблема:** есть `plan_inflow`, `plan_outflow`, но поле `deviation_pct` считается как `(fact - plan) / plan * 100`; при этом поля `fact` и `plan` не определены. Нужно считать отклонения отдельно по поступлениям, выплатам и чистому денежному потоку. Также нужна защита от plan = 0.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Добавить отдельные поля: <b>fact_inflow, fact_outflow, fact_net_cashflow, plan_net_cashflow, deviation_inflow_pct, deviation_outflow_pct, deviation_net_pct</b>.<br>
Формула: <b>deviation_pct = (fact_value - plan_value) / plan_value * 100</b>, если plan_value &ne; 0; если plan_value = 0, отклонение выводится как абсолютная сумма и помечается отдельным флагом.
</div>

## 8. Управленческий баланс: термин «пассивы» используется некорректно

**Где:** раздел 7.3.  
**Проблема:** сейчас `total_liabilities` назван «Пассивы, всего», а `equity = total_assets - total_liabilities`. В балансе пассивы = обязательства + капитал, а не только обязательства. Иначе в отчёте будут путаться «обязательства» и «пассивы».

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Переименовать:<br>
<b>total_liabilities → liabilities_total / Обязательства, всего</b>.<br>
Добавить поле:<br>
<b>equity → Собственный капитал = total_assets - liabilities_total</b>.<br>
Если нужен баланс в классическом виде, добавить:<br>
<b>passive_total = liabilities_total + equity</b>, контроль: <b>total_assets = passive_total</b>.
</div>

## 9. Источники для баланса могут отсутствовать в 1С:УТ

**Где:** раздел 7.3.  
**Проблема:** основные средства, собственный капитал, кредиты и займы часто ведутся в бухгалтерской базе, а не в 1С:УТ. Если исходный источник только УТ, управленческий баланс может быть неполным.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Добавить ограничение:<br>
<b>Баланс первой очереди является управленческим и строится по доступным данным УТ: ДС, товарные остатки, ДЗ/КЗ, взаиморасчёты. Основные средства, кредиты, займы и собственный капитал включаются только при наличии источника данных в УТ либо при подключении дополнительной бухгалтерской базы.</b>
</div>

## 16. Не хватает контрольных сверок и критериев приёмки

**Где:** весь документ, особенно разделы по отчётам.  
**Проблема:** документ описывает поля и визуализацию, но почти не описывает проверку корректности данных. Для заказчика это критично: отчёты должны сходиться с 1С и быть проверяемыми.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Добавить к каждому отчёту подраздел «Контрольные сверки и приёмка»:<br>
1. Сверка итогов отчёта с контрольным отчётом/выгрузкой из 1С за выбранный период.<br>
2. Проверка исключённых внутригрупповых оборотов отдельной таблицей.<br>
3. Проверка пустых/неразнесённых статей и справочников.<br>
4. Фиксация допустимого расхождения, например 0 руб. для денежных оборотов и согласованный порог для предварительных управленческих данных.<br>
5. Подтверждение отчёта ответственным представителем заказчика.
</div>
