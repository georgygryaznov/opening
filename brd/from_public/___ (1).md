# Новые замечания к файлу `Коммерческое_предложение.md`

Основание проверки:
- актуальный `opening_1step_reports_financial.md`;
- исходные `opening_Business.docx` и `opening_Reports.docx`;
- неизменённый файл `Коммерческое_предложение.md`.

Ниже — только актуальные замечания к коммерческому предложению после обновления технического проекта.

---

## 1. В цели КП нужно отразить новую архитектуру с DWH/витриной

**Где:** раздел 1, строка с целью: «Построить работающий ETL-процесс “1С → Битрикс24”...»

**Проблема:** в актуальном техническом проекте архитектура уже не прямая `1С → Битрикс24`, а `1С УТ 11.6 → ETL → DWH/витрина управленческих данных → Битрикс24`. В КП сейчас это не отражено.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Заменить цель на:<br><br>
<b>Построить работающий контур управленческой отчётности: 1С УТ 11.6 → ETL-процесс → DWH/витрина управленческих данных → Битрикс24, и последовательно реализовать 7 финансовых отчётов первой очереди.</b>
</div>

## 2. Нужно явно написать, что механизм Битрикс24 выбирается после полной проверки

**Где:** раздел 1 и этап 0.

**Проблема:** актуальный технический проект фиксирует, что механизм хранения, загрузки и визуализации в Битрикс24 будет выбран только после полной проверки редакции, модулей, прав, лимитов REST API и объёма данных. В КП пока звучит так, будто таблицы/поля в Битрикс24 уже точно известны.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Добавить в раздел 1:<br><br>
<b>Конкретный механизм реализации отчётов в Битрикс24 будет выбран после полной технической проверки редакции Битрикс24, доступных модулей, прав администратора, лимитов REST API и требований к объёму данных.</b>
</div>

## 3. Этап 0 должен начинаться с обследования, а не сразу с настройки Python ↔ 1С

**Где:** этап 0, состав работ.

**Проблема:** в актуальном техническом проекте многие решения зависят от проверки: способ интеграции с 1С, механизм Битрикс24, состав объектов 1С, источники расходов/баланса. Поэтому коммерчески безопаснее сначала продать этап 0 как архитектуру + обследование + пилот, а не как немедленную разработку.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
В начало состава работ этапа 0 добавить:<br>
1. Проверка доступа к 1С УТ 11.6 и доступных способов выгрузки данных.<br>
2. Проверка редакции Битрикс24, доступных модулей, прав и лимитов API.<br>
3. Проверка состава документов, регистров и справочников 1С для первой очереди.<br>
4. Согласование финальной схемы: 1С → ETL → DWH/витрина → Битрикс24.<br>
5. Только после этого — настройка пилотного ETL и пилотного отчёта.
</div>

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">

<b>Первичная DWH/витрина управленческих данных для пилотного отчёта: очищенные и агрегированные данные, подготовленные для загрузки/отображения в Битрикс24.</b>
</div>

</div>

## 7. В ОПиУ нужно убрать намёк на менеджерский результат

**Где:** этап 1.

**Проблема:** в актуальном техническом проекте менеджерский результат из ОПиУ убран. В текущем КП прямо он не указан, но лучше не добавлять его в методику и не смешивать с ОПиУ.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
В методической документации по ОПиУ описывать только управленческий P&amp;L: выручка, себестоимость, валовая прибыль, расходы, EBITDA, чистая прибыль, Contribution Margin и исключение внутригрупповых оборотов. Менеджерский результат по оплатам не включать в этап ОПиУ.
</div>

## 10. Dashboard первой очереди всё ещё содержит блоки будущих очередей

**Где:** этап 7, пункты 3–4.

**Проблема:** в неизменённом КП указаны Unit-экономика, ABC-анализ и логистика. В актуальном техническом проекте Dashboard первой очереди ограничен финансовым контуром: ОПиУ, ОДДС, баланс, дебиторка, платёжный календарь, план-факт. Unit-экономика, ABC и логистика относятся к следующим очередям.

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
<b>Предлагаемая правка:</b><br>
Заменить состав Dashboard на:<br>
1. 8 KPI-карточек: выручка, валовая прибыль, чистая прибыль, остаток ДС, Cash Flow недели, дебиторская задолженность, кредиторская задолженность, план-факт расходов.<br>
2. 3 графика: выручка/прибыль, Cash Flow, план-факт ключевых отклонений.<br>
3. 3 управленческие таблицы: юрлица, дебиторская задолженность по срокам, ближайшие платежи и риски кассовых разрывов.<br><br>
<b>Unit-экономику, ABC-анализ, склад и логистику указать как будущие блоки следующих очередей, не как результат первой очереди.</b>

<div style="border: 2px solid #d00000; color: #d00000; padding: 12px; border-radius: 6px; margin: 12px 0;">
